# Examples

These scenarios are imported from [`cloudflare/workers-sdk`](https://github.com/cloudflare/workers-sdk) fixtures and curated to exercise the binding shapes that matter for `workers-native`. They live under [`examples/upstream/`](./upstream) and are regenerated by:

```bash
bun run scripts/sync-examples.ts
```

The pinned commit and curated subset are tracked in [`examples/upstream.lock.json`](./upstream.lock.json). Re-running the sync against the same lock file produces the same tree; bumping a commit is a deliberate, reviewable change to the lock file plus a fresh sync.

The matrix below describes the **binding behavior** each fixture is meant to verify — not how `workers-native` implements it. Where the upstream fixture expects multiple Wrangler configs in one project, the exact `--config` / `--env` / `--native-config` flag is shown.

## Verification matrix

| # | Fixture (under `examples/upstream/`) | Binding shape | Expected with `workers-native` | Verify |
| - | --- | --- | --- | --- |
| 1 | `durable-objects-app` | Same-worker Durable Object binding (`MY_DO` → `FooBar`) | **Working** | `bunx workers-native serve --project ./examples/upstream/durable-objects-app` then `curl localhost:8787` |
| 2 | `unbound-durable-object` | DO class declared in `migrations` but **no** `durable_objects.bindings` entry (worker still starts cleanly) | **Working** | `bunx workers-native serve --project ./examples/upstream/unbound-durable-object` then `curl localhost:8787` |
| 3 | `dev-registry` (internal DO) | DO bound to a class in the **same** script — `wrangler.internal-durable-object.jsonc` | **Working** | `bunx workers-native serve --project ./examples/upstream/dev-registry --config ./examples/upstream/dev-registry/wrangler.internal-durable-object.jsonc` then `curl localhost:8787` |
| 4 | `dev-registry` (external DO via `script_name`) | DO binding with `script_name: internal-durable-object`, which lives in a **different** wrangler config — `wrangler.external-durable-object.jsonc` | **Expected failure** — startup succeeds; DO method calls fail because the auxiliary script is not auto-surfaced from another Wrangler file | `bunx workers-native serve --project ./examples/upstream/dev-registry --config ./examples/upstream/dev-registry/wrangler.external-durable-object.jsonc` then `curl localhost:8787` |
| 5 | `dev-registry` (nonexistent DO class) | DO binding with `script_name: internal-durable-object` and a class (`NonExistentObject`) that does not exist in the target — `wrangler.nonexistent-durable-object.jsonc` | **Expected failure** — DO class cannot be resolved | `bunx workers-native serve --project ./examples/upstream/dev-registry --config ./examples/upstream/dev-registry/wrangler.nonexistent-durable-object.jsonc` then `curl localhost:8787` |
| 6 | `dev-registry` (nonexistent entrypoint) | `[[services]]` binding pointing at `entrypoint: NonExistentEntrypoint` on `worker-entrypoint` — `wrangler.nonexistent-entrypoint.jsonc` | **Expected failure** — startup succeeds; `env.NAMED_ENTRYPOINT.fetch(...)` fails at request time | `bunx workers-native serve --project ./examples/upstream/dev-registry --config ./examples/upstream/dev-registry/wrangler.nonexistent-entrypoint.jsonc` then `curl localhost:8787` |
| 7 | `workers-with-assets-and-service-bindings` (caller `workerA`) | Four `[[services]]` bindings (`DEFAULT_EXPORT`, `DEFAULT_ENTRYPOINT`, `NAMED_ENTRYPOINT`, `WS`) pointing at sibling workers that are **not** declared as `[[workers]]` auxiliaries | **Expected failure** — startup succeeds; cross-worker calls fail until the callees are registered via `worker-native.toml` `[[extra_workers]]` or programmatic `extraWorkers` | `bunx workers-native serve --project ./examples/upstream/workers-with-assets-and-service-bindings/workerA --config ./examples/upstream/workers-with-assets-and-service-bindings/workerA/wrangler.jsonc` then `curl localhost:8787` |
| 8 | `worker-with-resources` (phase 2) | Mixed-resource bindings: KV, D1, R2, Workflows, internal DO (`DO`), **and** cross-script DO `DO_B` with `script_name: worker-b` | **Expected failure** — the primary starts and all local resource bindings (KV/D1/R2/internal DO) work; `env.DO_B.*` fails because the `worker-b` sibling is not auto-surfaced from its separate Wrangler config | `bunx workers-native serve --project ./examples/upstream/worker-with-resources --config ./examples/upstream/worker-with-resources/wrangler.jsonc` then `curl localhost:8787` |

> Tip — to bypass the request-time failure in row 7, declare each callee in a `worker-native.toml` at the project root and pass `--native-config <path>`. Example:
>
> ```toml
> [[extra_workers]]
> name = "worker-b"
> wrangler_project_root = "../workerB-with-default-export"
> ```
>
> Same applies to row 8 (`worker-b`).

## Notes

- These fixtures are **read-only** in this repo. Edit [`examples/upstream.lock.json`](./upstream.lock.json) and re-run the sync script to change them.
- "Expected failure" rows are the curated negative-path scenarios; the test suite enforces that those rows stay explicitly marked so behavior drift is caught quickly.
- Outcomes are about **binding resolution semantics** — not about how `workers-native` is implemented internally. See the root [`README.md`](../README.md) for the implementation notes (Wrangler `unstable_*` integration, `worker-native.toml`, `extraWorkers` precedence).
